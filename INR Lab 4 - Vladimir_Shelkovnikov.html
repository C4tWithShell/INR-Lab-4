Deploy OSPF

First of all, I created the loopback interface, which I select as the OSPF router ID.

interface bridge add name=loopback arp=enabled
ip address add address=10.255.255.<id>/32 interface=loopback
routing ospf instance edit 0 router-id <ip>

<id> - number of router ( Router1 - 1, Router2 - 2 and etc).
<ip> - IP of the loopback interface for router (Router1 - 10.255.255.1/32, Router2 - 10.255.255.2/32 and etc).

Which interface you will select as the OSPF router ID and why?

The router ID is required to uniquely identify a specific router. We can use the IP address of the active physical interface, but it is better to use the loopback interface because it is always up.

Then I moved on to setting up OSPF networks for each router.
routing ospf network add area=backbone network=<subnet>

<subnet> - 10.10.10.0/24 and 10.10.100.0/24 for Router1; 10.10.20.0/24 10.10.100.0/24 for Router2 and etc.

This completes the OSPF configuration. Then interfaces should appear on the OSPF Interfaces tab, and Routes should appear on the Routes tab.

What is the difference between advertising all the networks VS manual advertising (per interface or per subnet)? Which one is better?

With the help of advertising all the networks, we announce routes that take into account the parameters of each router. This can be inconvenient with a large network. Manual advertising allows you to achieve more optimal use of routes.

Which router is selected as DR and which one is BDR?

Each router has its own dr and bdr. DR and BDR were selected in OSPF according to the following criteria:

  1. DR: The router with the highest priority of the OSPF interface.
  2. BDR: The router with the second highest priority of the OSPF interface.
  3. If the priorities of the OSPF interfaces are equal, the highest router ID is used to make the selection.

DR (dedicated router) - a router that manages the messaging process in the OSPF network.
BDR (backup dedicated router) — a router that replaces the DR in case the latter is eliminated.

Multi-Area network and route summarization

Figure 12 shows the modified topology with 3 areas: area_20 (area x), area_10 (area Y), and backbone.


Figure 12 - Modified topology

First, I configured the External router as shown below.

//Configuring interfaces
ip address add address=172.13.100.1/24 interface=ether1
ip address add address=172.12.60.1/24 interface=ether2
ip address add address=172.12.10.1/24 interface=ether3

//Connecting area_10 to the backbone.
routing ospf network add area=backbone network=172.13.100.0/24

//Creating and configuring a new area - area_10
routing ospf area add name=area_10 area-id=0.0.0.10
routing ospf network add network=172.12.10.0/24 area=area_10
routing ospf network add network=172.12.60.0/24 area=area_10

//Adding route summarization
routing ospf area range add advertise=yes area=area_10  range=172.12.0.0/16

After that, I configured the Second router:

//Configuring interfaces
ip dhcp-client add interface=ether1
ip address add address=172.12.10.2/24 interface=ether2
ip address add address=172.12.80.1/24 interface=ether3

//Creating and configuring a new area - area_20
routing ospf area add name=area_20 area-id=0.0.0.20
routing ospf network add network=172.12.80.0/24 area=area_20

I have configured the last router:

ip address add address=172.13.100.2/24 interface=ether1
routing ospf network add network=172.13.100.0/24 area=backbone

After adding the route summarization, the subnets 172.12.60.0/24, 172.12.10.0/24, and 172.12.80.0/24 were merged into 172.12.0.0/16 (Figure 13). But i can ping them.


Figure 13 - Route summarization

Why every area has to be connected to area 0 ?

In OSPF domains, the topology of the area is limited, so there must be a backbone area (area 0), and all other areas must have physical or virtual connections to the backbone.
The reason for this is that routing between OSPF regions uses a distance vector approach, and a strict hierarchy of regions avoids the problem of counting to infinity.
In order to overcome this limitation, the protocol provides a virtual link tool (virtual link).

Area X (area_20)

To create a virtual link to area_20, I configured the routers as shown below.

External:
routing ospf virtual-link add transit-area=area_10 neighbor-id=4.4.4.4

Second:
routing ospf virtual-link add transit-area=area_10 neighbor-id=1.1.1.1
